## Core Architecture & Subsystems

FlatCAM Evo is a PyQt6 desktop application that turns PCB design files (Gerber, Excellon, SVG, DXF, PDF, HPGL2) into machine-ready G-Code. Its architecture is organized around a single central hub (`App`), a GUI-free CAM engine (`camlib`), a swappable graphics layer, and a background worker pool, all wired together with Qt signals.

### The App hub (`appMain.py`)

The `App` class (~9000 lines) is the heart of the application — nearly every subsystem hangs off `self.app`. It owns:

- **The object collection** — `self.collection = ObjectCollection(app=self)`, the Qt tree model that holds every project object.
- **Tool/plugin instances** — all ~30 plugins from `appPlugins/` are instantiated directly in `App.__init__` (not dynamically discovered) and register themselves into the menu/notebook.
- **The canvas** — `self.plotcanvas`, plus utility `ShapeCollection`s (`tool_shapes`, `hover_shapes`, `sel_shapes`) used for selection/hover/tool overlay geometry.
- **The worker crew** — `self.workers = WorkerStack(...)`, fed via the `self.worker_task` signal.
- **App lifecycle and signals** — object creation/plotting, file I/O dispatch, mouse/keyboard event routing, preferences/defaults, and the Tcl shell.

A key runtime switch lives here: `self.use_3d_engine` selects the rendering backend at startup (defaulting to VisPy, falling back to matplotlib). Much of `App`'s event-handling code branches on this flag because mouse coordinates and canvas APIs differ between backends (`event.pos` vs `(event.xdata, event.ydata)`, right-button index 2 vs 3, `plotcanvas.update()` vs `auto_adjust_axes()`, etc.).

### The CAM engine (`camlib.py`)

`camlib.py` is the GUI-free geometry and toolpath core. It is pure model/algorithm code with no Qt dependencies, built on Shapely (>= 2.0). Key classes:

- **`Geometry`** — the base model for 2D geometry: holds Shapely geometry, performs buffering/isolation/offsetting, paint/clear operations, and the Shapely-based algorithms that underpin most tools.
- **`CNCjob(Geometry)`** — extends `Geometry` to generate and hold G-Code: toolpath planning, feed/plunge handling, and emission of machine code through the preprocessor layer.
- **`ApertureMacro` / `ParseError`** — Gerber aperture-macro expansion and parse error signaling.
- **`AppRTree` / `AppRTreeStorage`** — spatial indexing structures used to accelerate geometry lookups.

The GUI object wrappers in `appObjects/` delegate the heavy CAM work down to these classes; G-Code dialect formatting is delegated further to `preprocessors/`.

### Dual rendering backends and ShapeCollection

Objects never draw to the screen directly. Instead they populate a backend-neutral **`ShapeCollection`**, and the active canvas renders it. Two interchangeable backends exist:

- **VisPy / OpenGL (default)** — `appGUI/PlotCanvas.py`. `PlotCanvas` mixes `QtCore.QObject` with `VisPyCanvas` and uses GPU-accelerated VisPy visuals (`ShapeCollection`, `ShapeGroup`, `TextCollection`, `Cursor`, `Line`, `Rectangle`, etc.). VisPy is monkeypatched at startup via `appGUI/VisPyPatches.py`.
- **Matplotlib legacy fallback** — `appGUI/PlotCanvasLegacy.py`. Uses a matplotlib `Figure`/`FigureCanvasQTAgg` with a `CanvasCache` (renders to an off-screen bitmap on a background request, then blits to the visible canvas) and `ShapeCollectionLegacy`. Solid polygons are drawn via the vendored `descartes` Shapely→matplotlib patch package.

Because of this split, any rendering-affecting change must be implemented for both backends. The two `ShapeCollection` implementations expose parallel APIs so callers stay backend-agnostic; the VisPy variant can offload geometry triangulation to a multiprocessing pool, while tiny per-frame utility geometry (selection rectangle, hover) is deliberately kept inline to avoid IPC/pickle overhead during drags.

### Worker thread pool (`appWorkerStack.py`)

GUI code must never be touched from a worker thread. Long-running operations are dispatched through `WorkerStack`, a fixed-size crew of `Worker` QObjects (count from `global_worker_number`, default 2), each moved onto its own `QThread`. Work is submitted by emitting `app.worker_task` with a dict `{'fcn': callable, 'params': [...]}`; `WorkerStack.add_task` load-balances by assigning the task to the least-busy worker (tracked in `self.load`) and re-emits it as a `worker_task` signal the worker picks up. Results flow back to the GUI exclusively via Qt signals (e.g. `task_completed`, `thread_exception`). `FCProcess`/`FCProcessContainer` (`appProcess.py`) wrap operation lifecycle for status-bar feedback.

### Overall layer structure

- **`appObjects/`** — the object model. `FlatCAMObj` (in `AppObjectTemplate.py`) is the base for `GerberObject`, `ExcellonObject`, `GeometryObject`, `CNCJobObject`, `ScriptObject`, `DocumentObject`. Objects are created through the `AppObject.new_object(kind, name, initialize)` factory, which looks the class up in a kind-keyed `classdict` and copies the matching `<kind>_`-prefixed app defaults into the object's `obj_options` (a `LoudDict` with change callbacks that drive reactive UI updates).
- **`appParsers/`** — file-format parsers (Gerber, Excellon, SVG, DXF, PDF, HPGL2, fonts). Parsing logic lives here, not in `appObjects`; Gerber/Excellon objects multiply-inherit from both `FlatCAMObj` and their parser class.
- **`appGUI/`** — the Qt presentation layer: `MainGUI` (QMainWindow), custom widgets in `GUIElements.py` (FCEntry, FCDoubleSpinner, RadioSet, …) used in place of raw Qt widgets, the preferences UI under `appGUI/preferences/`, themes, and both canvas backends.
- **`appHandlers/`** — dispatch layer: `appIO.py` (file open/save) and `appEdit.py` (editor entry points).
- **`appEditors/`** — interactive editors per object type (GeoEditor, GerberEditor, ExcEditor, GCodeEditor, TextEditor), each with its own plugin subfolder.

Supporting subsystems include `appPlugins/` (the ~30 `AppTool` tools), `preprocessors/` (auto-registered G-Code dialect post-processors), `tclCommands/` (auto-imported Tcl shell commands), and `defaults.py`/`appDatabase.py` for settings and the tools database. Data flow is typically: a parser builds geometry → wrapped in an `appObjects` model → CAM operations run in `camlib` on a background `WorkerStack` thread → results pushed into a `ShapeCollection` → rendered by the active canvas backend, with all cross-thread communication mediated by Qt signals.

## Object Types & Editors

FlatCAM's data model is built around six object kinds, all subclassing `FlatCAMObj` (`appObjects/AppObjectTemplate.py`). Every object is created through the factory `AppObject.new_object(kind, name, initialize, ...)` (`appObjects/AppObject.py`), which:

1. Looks up the concrete class in a `classdict` keyed by the kind string (`'gerber'`, `'excellon'`, `'cncjob'`, `'geometry'`, `'script'`, `'document'`).
2. Instantiates it and copies every app default prefixed `<kind>_` into the object's `obj_options` (a `LoudDict` with change callbacks), stripping the prefix. Manufacturing kinds additionally inherit `tools_mill_*` / `tools_*` options; Excellon also gets `tools_drill_*`, Gerber also gets `tools_iso_*`.
3. Runs the caller-supplied `initialize(obj, app)` (parsing/geometry build, possibly on a worker thread).
4. Converts units if needed, computes the bounding box (`xmin/ymin/xmax/ymax`, skipped for script/document), moves the object to the main thread, and emits `object_created`. `on_object_created` appends it to the `ObjectCollection`, assigns plot colors, selects it, and threads the initial plot.

The base `FlatCAMObj` provides the common machinery: `obj_options`/form binding (`to_form`/`read_form`/`set_form_item`/`read_form_item`), plotting into a `ShapeCollection` (VisPy) or `ShapeCollectionLegacy` (matplotlib) via `add_shape`/`clear`/`visible`, the Properties tree (`add_properties_items`/`update_area_chull` computing dimensions, box/convex-hull/copper area, routing time, etc.), name change handling (`on_name_activate`), generic transforms wired from the common UI (`on_offset_button_click`, `on_scale_button_click`, `on_skew_button_click`), and (de)serialization (`from_dict`).

`AppObject` also exposes convenience constructors for empty objects: `new_gerber_object`, `new_excellon_object`, `new_geometry_object` (single tool, multitool/multigeo), `new_script_object` (seeds a Tcl template), and `new_document_object`.

### Object Types

#### Gerber (`GerberObject`, kind `"gerber"`)
- **Represents:** a parsed Gerber layer — copper traces, pads, regions. Holds `solid_geometry`, optional `follow_geometry`, and an aperture dict `self.tools` (each aperture id maps to type/size and per-element `solid`/`follow`/`clear` geometry). `multitool=False`, `multigeo=False`.
- **Created by:** opening a `.gbr` file (parser inheritance from the Gerber parser in `appParsers/`), `new_gerber_object`, the Gerber editor, or Tcl. A Gerber X2 drill file is auto-converted to Excellon.
- **Operations:**
  - **Isolation routing** — `isolate()` / `generate_envelope()`: generates isolation tool-paths (single/multi-pass, overlap, combine) producing a Geometry object.
  - **Follow geometry** — `follow_geo()`: traces aperture centerlines.
  - **Non-copper regions** — `on_generate_non_copper_button_click()`: the inverse area for clearing.
  - **Bounding box** — `on_generatebb_button_click()`: a margined boundary geometry.
  - **Buffer** — `on_generate_buffer()` / `buffer()`.
  - **Export** — `export_gerber()` (re-emits Gerber RS-274X with given digits/zero-suppression/scale).
  - **Transforms** — `mirror`, `offset`, `rotate`, `scale`, `skew`.
  - **Plotting/visualization** — `plot()`, per-aperture marking (`plot_aperture`, `on_mark_cb_click_table`, `on_mark_all_click`, `clear_mark_all`), multicolor/solid/follow toggles, aperture table.
  - **Merge** of multiple Gerbers (`merge`), unit conversion (`convert_units`), editing via the Gerber Editor.

#### Excellon (`ExcellonObject`, kind `"excellon"`)
- **Represents:** an NC drill/route file. `self.tools` maps each tool to `tooldia`, a list of `drills` (points) and `slots` (segment endpoints), plus per-tool `data` and `solid_geometry`.
- **Created by:** opening an Excellon file (parser inheritance), `new_excellon_object`, the Excellon editor, Gerber-X2-drill conversion, or Tcl.
- **Operations:**
  - **Drilling CNCJob** — produces drilling G-Code (via the Drilling plugin) for selected tools.
  - **Mill drills / mill slots** — `generate_milling_drills()` / `generate_milling_slots()` (`on_generate_milling_button_click`, `on_generate_milling_slots_button_click`): convert holes/slots into milled Geometry using a chosen mill tool.
  - **Export** — `export_excellon()` (digits/zero-suppression, decimal vs leading/trailing, slot type).
  - **Tool table** UI: row selection, toggle rows, copy diameter, autoload from Tools DB.
  - **Transforms** (inherited), `merge` (with optional tool fusing, decimal alignment), `convert_units`, multicolor/solid plot toggles, editing via the Excellon Editor.

#### Geometry (`GeometryObject`, kind `"geometry"`)
- **Represents:** generic 2D tool-path geometry (often the result of isolation/cutout/paint/NCC, or imported DXF/SVG). Can be `multigeo` (per-tool geometry) and `multitool`. `self.tools` carries per-tool `tooldia`, `offset`/`offset_value`, `type`, `tool_type`, `data`, and `solid_geometry`.
- **Created by:** Gerber isolation/non-copper/bbox, cutout/paint/NCC tools, DXF/SVG import, `new_geometry_object`, the Geometry editor, or Tcl.
- **Operations:**
  - **CNCJob generation** — `mtool_gen_cncjob()` (multi-tool) and `generatecncjob()` (single tool): the core milling path → G-Code step (cut/move Z, feedrate, multidepth, tool change, etc.).
  - **Export DXF** — `export_dxf()`.
  - **Geometry utilities** — `on_simplify_geometry()`, `on_calculate_vertex_points()`.
  - **Transforms** — `scale`, `offset` (plus mirror/rotate/skew through common UI), `convert_units`.
  - **Tool table** management (`on_row_changed`, `table_toggle_all`, `select_tools_table_row`), `merge` (multi-geo/fuse aware), per-tool plotting, editing via the Geometry Editor.

#### CNCJob (`CNCJobObject`, kind `"cncjob"`)
- **Represents:** generated machine code. Subclasses `CNCjob` from `camlib.py`; carries `gcode`, `gcode_parsed` (toolpath geometry for plotting), `routing_time`, `travel_distance`, and `obj_options['type']` of `'geometry'` or `'excellon'` recording its source.
- **Created by:** `GeometryObject.generatecncjob`/`mtool_gen_cncjob` or Excellon drilling; not loaded directly (machine code is imported as text/script).
- **Operations:**
  - **Export G-Code** — `on_exportgcode_button_click()` / `export_gcode()` / `export_gcode_handler()` (with preamble/postamble, preprocessor dialect), `get_gcode()`.
  - **Review/edit code** — `on_review_code_click()` opens the GCode Editor; `on_update_source_file()` syncs edits back.
  - **SVG export** — `get_svg()`.
  - **G-code header/footer** assembly (`gcode_header`, `gcode_footer`).
  - **Plot controls** — `plot(kind='all'|'travel'|'cut')`, `on_plot_kind_change`, `on_updateplot_button_click`, travel/cut annotation toggles (`on_annotation_change`), CNC-tools tables (`build_cnc_tools_table`, `build_excellon_cnc_tools`), `convert_units`. (No geometric transforms.)

#### Script (`ScriptObject`, kind `"script"`)
- **Represents:** a Tcl script (a text buffer, `source_file`); no geometry. Uses the Code/Text editor as its UI.
- **Created by:** `new_script_object` (with a starter template), opening a `.FlatScript`/text file (`parse_file`), or Tcl.
- **Operations:** **Run code** (`handle_run_code` — executes the script in the in-app Tcl shell), autocomplete toggle, file parsing. Geometric methods (`mirror/offset/rotate/scale/skew/buffer/bounds`) are no-op stubs so the object satisfies the common interface.

#### Document (`DocumentObject`, kind `"document"`)
- **Represents:** a rich-text notes document (no geometry), edited inline.
- **Created by:** `new_document_object` or Tcl.
- **Operations:** rich-text formatting — bold/italic/underline (`on_bold_button`, `on_italic_button`, `on_underline_button`), font family/size, font and selection colors, tab size, read-only toggle, autocomplete. Geometric methods are no-op stubs.

### Editors

Editors are activated from the object's context/menu (dispatched via `appHandlers/appEdit.py`). Each builds a toolbar/palette of "draw tools" and, on commit, writes the modified geometry back into the source object. The Geometry, Gerber, and Excellon editors share a draw-tool architecture: a `tools*` dict maps a tool key to a toolbar button and a constructor class; `make_callback`/`on_tool_select` activate the chosen tool, and each tool subclasses a `DrawTool`/`FCShapeTool` (or `*EditorGrb`/`*EditorExc`) base. Many tools have dedicated parameter panels under the plugin subfolders (`geo_plugins/`, `grb_plugins/`, `exc_plugins/`).

#### Geometry Editor — `AppGeoEditor` (`appEditors/appGeoEditor.py`)
Edits **Geometry** objects (shape-level editing of Shapely geometry). Tool palette (`connect_geo_toolbar_signals`):
- **select** (`FCSelect`) — pick/move/marquee-select shapes.
- **circle** (`FCCircle`), **arc** (`FCArc`), **rectangle** (`FCRectangle`), **polygon** (`FCPolygon`), **path** (`FCPath`) — primitive drawing tools.
- **text** (`FCText`) — render text as geometry.
- **buffer** (`FCBuffer`) — buffer/inflate selected shapes (panel `GeoBufferPlugin`).
- **simplification** (`FCSimplification`) — reduce vertices (panel `GeoSimplificationPlugin`).
- **paint** (`FCPaint`) — fill/clear a polygon area (panel `GeoPaintPlugin`).
- **eraser** (`FCEraser`) — subtract geometry.
- **move** (`FCMove`) and **copy** (`FCCopy`) — relocate/duplicate selections (panel `GeoCopyPlugin`).
- **explode** (`FCExplode`) — break a polygon into its boundary lines.
- **transform** (`FCTransform`) — rotate/skew/scale/mirror/offset (panel `GeoTransformationPlugin`).

Additional plugin panels available: path drawing (`GeoPathPlugin`), circle/rectangle parameter panels (`GeoCirclePlugin`, `GeoRectanglePlugin`), text (`GeoTextPlugin`), and **stitch** (`GeoStitchPlugin`, joining open line ends). `activate`/`deactivate` swap the editor in/out, hiding the source object's plot while editing; `on_transform_complete`/`plot_all` refresh the canvas; grid entries (`on_gridx_val_changed`, `on_gridy_val_changed`) drive snapping.

#### Gerber Editor — `AppGerberEditor` (`appEditors/appGerberEditor.py`)
Edits **Gerber** objects at the aperture/flash/track level. Tool palette (`tools_gerber`):
- **select** (`SelectEditorGrb`).
- **pad** (`PadEditorGrb`) and **array** (`PadArrayEditorGrb`) — place single pads / linear & circular pad arrays (panels `GrbPadPlugin`, `GrbPadArrayPlugin`).
- **track** (`TrackEditorGrb`) — draw copper tracks (panel `GrbTrackPlugin`).
- **region** (`RegionEditorGrb`) — filled copper regions (panel `GrberRegionPlugin`).
- **poligonize** (`PoligonizeEditorGrb`) — convert selected geometry to a polygon.
- **semidisc** (`DiscSemiEditorGrb`) and **disc** (`DiscEditorGrb`) — half-disc / disc primitives.
- **buffer** (`BufferEditorGrb`, panel `GrbBufferPlugin`), **simplify** (`SimplifyEditorGrb`, panel `GrbSimplificationPlugin`), **scale** (`ScaleEditorGrb`).
- **markarea** (`MarkEditorGrb`) — highlight apertures by area.
- **import** (`ImportEditorGrb`) — import geometry into the Gerber.
- **eraser** (`EraserEditorGrb`), **copy** (`CopyEditorGrb`, panel `GrbCopyPlugin`), **move** (`MoveEditorGrb`).
- **transform** (`TransformEditorGrb`, panel `GrbTransformationPlugin`).

Aperture management is table-driven (add/remove/select apertures); `GrbCommon` holds shared helpers.

#### Excellon Editor — `AppExcEditor` (`appEditors/appExcEditor.py`)
Edits **Excellon** objects (drills and slots per tool). Tool palette (`tools_exc`):
- **drill_select** (`SelectEditorExc`).
- **drill_add** (`DrillAdd`) and **drill_array** (`DrillArray`) — add single drills / linear & circular drill arrays (panels `ExcDrillPlugin`, `ExcDrillArrayPlugin`).
- **slot_add** (`SlotAdd`) and **slot_array** (`SlotArray`) — add single slots / slot arrays (panels `ExcSlotPlugin`, `ExcSlotArrayPlugin`).
- **drill_resize** (`ResizeEditorExc`) — change tool diameter (panel `ExcResizePlugin`).
- **drill_copy** (`CopyEditorExc`, panel `ExcCopyPlugin`) and **drill_move** (`MoveEditorExc`) — duplicate/relocate selected drills/slots.

Geometry is kept in an `AppRTreeStorage` (`make_storage`); the tool table manages diameters and the holes assigned to each tool.

#### GCode Editor — `AppGCodeEditor` (`appEditors/appGCodeEditor.py`)
Edits the machine code of a **CNCJob** object. `edit_fcgcode(cnc_obj)` loads `cnc_obj.source_file` into the embedded Code Editor (an `AppTextEditor` tab); `update_fcgcode()` writes the edited text back to `source_file`. Provides a CNC-tools table (separate layouts for Geometry-sourced vs Excellon-sourced jobs) with per-tool row selection (`on_row_selection_change`, `on_toggle_all_rows`) to navigate/scope the code, editable **CNC code snippets** (prepend/append blocks defined in Preferences), Beginner/Advanced level toggle (`on_level_changed`), open external G-code (`on_open_gcode`), and object renaming. Actual text manipulation (find/replace/save) is delegated to the text-editor tab.

#### Text Editor — `AppTextEditor` (`appEditors/appTextEditor.py`)
The shared code/text editing widget used by **Script** and **Document** objects and embedded by the GCode Editor. Operations: `load_text` (plain or HTML), `on_text_changed`/`handleTextChanged` (dirty tracking), find and replace (`handleFindGCode`, `handleReplaceGCode`), open (`handleOpen`) and save (`handleSaveGCode`, with optional callback), and print/preview (`handlePrint`, `handlePreview`). A `set_callback` hook lets the host (e.g. the GCode editor) react to save.

## Plugins / Tools

All tools live in `appPlugins/`, inherit `AppTool` (`appTool.py`), are instantiated in `App.__init__`, register a menu action via `install()`, and swap their UI into the notebook's plugin tab on `run()`. Each tool's user-facing name is its `pluginName` attribute.

### Isolation, Copper & Trace Tools

- **ToolIsolation** (`Isolation`) — Generates isolation-routing geometry around the copper traces of a Gerber object by offsetting one or more passes of a chosen tool diameter, producing the toolpaths that mill away copper to electrically isolate tracks/pads.
- **ToolNCC** (`NCC` — Non-Copper Clearing) — Clears all copper from the non-track areas of a Gerber (the "copper pour" regions), producing geometry to remove unwanted copper across the whole board surface using one or more tools.
- **ToolPaint** (`Paint`) — Fills/empties the interior of selected polygons or areas with overlapping tool passes (lines, seed, or contour strategies), used to "paint" out copper inside a region.
- **ToolCopperThieving** (`Copper Thieving`) — Adds filler copper (solid, dots, squares, or lines pattern) to empty board areas to balance copper density for etching/plating, and can also generate a robber bar and pattern-plating mask.
- **ToolFollow** (`Follow`) — Creates "follow" geometry that traces the centerlines of Gerber traces (the path the aperture followed) rather than isolating around them, useful for single-line engraving/following toolpaths.
- **ToolEtchCompensation** (`Etch Compensation`) — Compensates Gerber copper features for the lateral undercut of chemical etching by growing/shrinking geometry based on copper thickness and etchant factor, so the etched result matches the intended dimensions.

### Drilling & Milling

- **ToolDrilling** (`Drilling`) — Generates drilling G-Code from an Excellon object: selects drill tools, sets feedrates/depths/peck/dwell and other drill parameters, and produces the CNCJob for drilling holes.
- **ToolMilling** (`Milling`) — Generates milling G-Code from Geometry (or mills holes/slots from Excellon), turning geometry toolpaths into a CNCJob with cut depth, multi-depth, feedrate, spindle and travel settings.
- **ToolSolderPaste** (`SolderPaste`) — Generates dispensing toolpaths/G-Code for applying solder paste to SMD pads, driving a paste-dispenser instead of a cutting tool.

### Cutout, Panelize & Board Layout

- **ToolCutOut** (`Cutout`) — Creates the board-perimeter cutout geometry to separate the finished PCB from the surrounding material, supporting freeform/rectangular cutouts and leaving holding "bridge gaps" (mouse-bites).
- **ToolPanelize** (`Panelize PCB` / `Panelization`) — Replicates a source object into a rows×columns panel/array with configurable spacing, producing a single panelized object for batch manufacturing.
- **ToolDblSided** (`2-Sided`) — Assists two-sided PCB fabrication by mirroring objects about an axis and computing alignment/bounding-box and drill reference points so the bottom layer registers with the top.
- **ToolAlignObjects** (`Align Objects`) — Aligns one object to another by picking matching reference points (e.g. for overlaying top and bottom layers), computing the needed translation/rotation.

### Gerber/Excellon Editing & Conversion

- **ToolExtract** (`Extract`) — Extracts drill holes (as an Excellon) and/or soldermask geometry from a Gerber object based on its pad shapes and apertures.
- **ToolPunchGerber** (`Punch Gerber`) — Punches holes into the pads of a Gerber object (removing pad centers using fixed, proportional, or Excellon-derived diameters) so pads become annular rings.
- **ToolInvertGerber** (`Invert Gerber`) — Inverts a Gerber object's polarity, swapping copper and clear areas within its bounding box (negative-image generation).
- **ToolSub** (`Subtract`) — Subtracts one object's geometry from another (Gerber-from-Gerber or Geometry-from-Geometry), removing overlapping features to produce a difference object.
- **ToolTransform** (`Transformation`) — Applies geometric transforms — rotate, skew, scale, mirror/flip, offset, and buffer — to selected objects.
- **ToolMove** (`Move`) — Interactively moves selected objects on the canvas from a clicked origin point to a destination.

### Import / Export

- **ToolImage** (`Image Import`) — Imports a raster image (e.g. BMP/PNG) and traces it into a Gerber or Geometry object, mapping black/white pixels into PCB geometry.
- **ToolPDF** (`PDF Import Tool`) — Parses a PDF file and imports its vector content as Gerber and/or Geometry objects.
- **ToolPcbWizard** (`PcbWizard Import`) — Imports Excellon drill files produced by PCBWizard, pairing the .INF/.DRL data to reconstruct a proper Excellon object.
- **ToolFilm** (`Film`) — Produces a film/artwork export (positive or negative, SVG/PDF/PNG) of an object for photo-plotting or printing transparencies.
- **ToolLaser** (`Laser`) — Generates laser-engraving/cutting G-Code (laser-mode toolpaths with S-power control) and exports the job, targeting laser engravers rather than spindle machines.
- **ToolQRCode** (`QRCode`) — Generates a QR code and renders it as Gerber geometry that can be placed on the PCB (e.g. for serial numbers or links).

### Fabrication Aids & Markers

- **ToolFiducials** (`Fiducials`) — Adds fiducial alignment marks (circular/cross/chess pattern) to the board, plus optional copper/soldermask coverage, for pick-and-place and registration.
- **ToolMarkers** (`Markers`) — Places corner/alignment marker geometry on an object to aid registration and positioning.
- **ToolLevelling** (`Levelling`) — Performs auto-leveling / Z-height probing: generates a probing grid over the board and adjusts G-Code so the toolpath follows surface variations (compensating for non-flat material).

### Measurement & Analysis

- **ToolDistance** (`Distance`) — Interactive ruler that measures the distance (and dx/dy/angle) between two points clicked on the canvas.
- **ToolObjectDistance** (`Object Distance`) — Measures the minimum distance between two selected objects on the canvas.
- **ToolOptimal** (`Find Optimal`) — Analyzes a Gerber object to find the minimum distance between its geometry elements, helping pick the largest safe isolation tool diameter; lists the locations where that minimum (and other close) spacings occur.
- **ToolRulesCheck** (`Check Rules`) — Runs a design-rules check (DRC) across Gerber/Excellon layers — trace width, clearance, hole-to-hole, drill size, silk-to-pad, ring width, etc. — and reports violations.
- **ToolReport** (`Object Report`) — Displays a report of the selected object's properties (type, dimensions/bounding box, options, and other metadata).

### Utilities

- **ToolCalculators** (`Calculators`) — Provides PCB-related calculators, including V-shape engraving tool-tip cut-width vs. depth and electroplating area/current estimation.
- **ToolShell** (`Shell`) — The in-app Tcl command shell/console, exposing the `tclCommands/` scripting commands and command history for automating FlatCAM via text commands.

## Supported File Formats & Import/Export

FlatCAM Evo reads and writes a broad set of CAD/CAM file formats. File-format parsing lives in `appParsers/` (GUI-free, one module per format), while the open/save *dispatch* — file dialogs, threading, and object creation — lives in `appHandlers/appIO.py` (the `appIO` class). The actual import/export *codecs* are methods on the object/geometry classes (`camlib.py`, `appObjects/`), which `appIO` orchestrates.

### Parser modules (`appParsers/`)

| Module | Format | Produces |
| --- | --- | --- |
| `ParseGerber.py` | RS-274X Gerber | `GerberObject` geometry + apertures |
| `ParseExcellon.py` | Excellon NC drill | `ExcellonObject` tools (drills/slots) |
| `ParseSVG.py` | SVG vector | shapes → Geometry or Gerber |
| `ParseDXF.py` (+ `ParseDXF_Spline.py`) | AutoCAD DXF (via `ezdxf`) | shapes → Geometry or Gerber |
| `ParsePDF.py` | PDF vector content streams | Gerber (from strokes) and/or Excellon (from white fills) |
| `ParseHPGL2.py` | HPGL/2 plotter (`.plt`) | `GeometryObject` |
| `ParseFont.py` | TTF/OTF fonts | glyph geometry (used by the Text tool, not a file-open path) |

### Import formats

Each importer is dispatched from a `File` menu callback in `appIO`, queued onto the `WorkerStack` via `self.worker_task.emit({'fcn': ..., 'params': [...]})`, and builds an object through `app.app_obj.new_object(kind, name, obj_init, ...)`.

- **Gerber** — `on_file_open_gerber` → `open_gerber()`. Recognizes a very wide extension set (`.gbr .ger .gtl .gbl .gts .gbs .gtp .gbp .gto .gbo .gm1 .gml .gm3 .gko`, plus Protel, Eagle `.cmp/.sol/...`, OrCAD `.top/.bot/...`, Allegro `.art`, Mentor `.pho/.gdo`, `.outline`, etc.). Creates a **`GerberObject`** via `gerber_obj.parse_file()`. Partial-parse damage is surfaced as a `"defective"` warning.
- **Excellon / NC drill** — `on_file_open_excellon` → `open_excellon()`. Extensions `.drl .txt .xln .drd .tap .exc .ncd`. Parses tools, then `create_geometry()`, producing an **`ExcellonObject`**.
- **G-Code** — `on_file_open_gcode` → `open_gcode()`. Large extension list (`.nc .ncc .tap .gcode .cnc .ecs .fnc .dnc .ncg .gc .fan .fgc .din .xpi .hnc .h .i .ncp .min .gcd .rol .knc .mpr .ply .out .eia .sbp .mpf`, `.txt`). Reads raw G-code into a **`CNCJobObject`**, sniffs `Type:` (Geometry/Excellon), `Preprocessor:`, per-tool `Dia:`/`Z_Cut:` comment lines, then `gcode_parse()` + `create_geometry()`.
- **SVG** — `on_file_import_svg(type_of_obj)` → `import_svg()`. Imported as either a **`GeometryObject`** or a **`GerberObject`** depending on the chosen `type_of_obj` (defaults to geometry). Backed by `Geometry.import_svg()`.
- **DXF** — `on_file_import_dxf(type_of_obj)` → `import_dxf()`. Also targets **Geometry** (`Geometry.import_dxf_as_geo`, `camlib.py`) or **Gerber** (`ParseGerber.import_dxf_as_gerber`).
- **PDF** — `import_pdf()` hands off to the PDF plugin (`appPlugins/ToolPDF.py` `open_pdf()`, run via the worker). Vector content is split by drawing operator/colour: strokes become a **`GerberObject`**, white fills become an **`ExcellonObject`** (one or more objects per PDF page).
- **HPGL2** — `on_file_open_hpgl2` (`.plt`) → `open_hpgl2()`. Parsed by `appParsers.ParseHPGL2.HPGL2` into a **`GeometryObject`** (multigeo).
- **Image (raster)** — handled by `appPlugins/ToolImage.py` `import_image()` (filter `*.BMP *.PNG *.JPG *.JPEG`). Two modes: `'raster'` (pixel→polygon) and `'trace'` (vectorized via SVG). Produces a **`GeometryObject`** or **`GerberObject`**.
- **TCL Script** — `on_file_open_script` / `on_file_open_script_example` (`.FlatScript .tcl .txt`) → `open_script()`, creating a **`ScriptObject`** opened in the Code Editor. `on_file_run_cript()` instead loads and executes a script directly in the Tcl shell without creating an object.
- **Project** — `on_file_open_project` (`*.FlatPrj`) → `open_project()` (run on the GUI thread, not the worker, because it is not thread-safe). Parses the JSON (LZMA-decompressed if compressed), then rebuilds every saved object via `from_dict()`. `is_legacy_project_dict()` detects and migrates older projects (old `options`/`apertures`/`cnc_tools` keys vs. the current `obj_options`/`tools`).
- **Configuration / Preferences** — `on_file_open_config` (`*.FlatConfig`, `*.json`) loads a config file into the Code Editor; `on_import_preferences` (`*.FlatConfig`) loads settings into the `options` dict via `options.load()`.

### Export / Save formats

FlatCAM distinguishes **Save** (write the object's embedded original `source_file`, via `save_source_file()`) from **Export** (regenerate the file from the in-memory geometry, applying unit/precision settings).

- **Gerber** — *Save*: `on_file_save_gerber` (`.GBR/.GRB`) writes the stored source. *Export*: `on_file_export_gerber` → `export_gerber()` regenerates RS-274X from a `GerberObject` (builds `%FS`/`%MO` header, `%ADD` aperture defs for C/R/O types, then `obj.export_gerber()` body). Honours `gerber_exp_units/integer/decimals/zeros`.
- **Excellon** — *Save*: `on_file_save_excellon` (`.DRL/.TXT`) writes the source. *Export*: `on_file_export_excellon` → `export_excellon()` regenerates an Excellon file (`M48` header, units/zeros `LZ`/`TZ`, decimal vs. `ndec` format, slot-type handling) via `obj.export_excellon()`. Honours `excellon_exp_*` settings.
- **DXF** — `on_file_export_dxf` → `export_dxf()`; **Geometry objects only**. Uses `obj.export_dxf()` (`ezdxf` drawing) and `saveas()`.
- **SVG** — `on_file_export_svg` → `export_svg()`. Accepts Geometry, Gerber, CNCJob, or Excellon objects; calls each object's `export_svg()` and wraps it with an SVG header/footer (Y-axis flip for Inkscape compatibility).
- **G-Code** — exported from a **`CNCJobObject`** (`export_gcode()` on the CNCJob, driven by the chosen preprocessor in `preprocessors/`), not from an `appIO.on_file_export_*` menu entry; the IO layer's G-code path is the *import* side (`open_gcode`).
- **PNG** — `on_file_export_png()` saves a screenshot of the canvas. For the 3D/VisPy backend it grabs a framebuffer screenshot (`_screenshot()` → `write_png`); for the matplotlib legacy backend it calls `figure.savefig()`.
- **PDF (print)** — `on_file_save_objects_pdf()` → `save_pdf()`. Renders selected objects (or the whole workspace) to PDF by exporting each to SVG, recolouring, composing one SVG, and rasterizing via `svglib`/`reportlab`. Supports a large paper-size table (A/B/C series, LETTER/LEGAL/LEDGER, etc.) plus a `Bounds` (fit-to-content) mode and portrait/landscape orientation.
- **Project** — `on_file_save_project` / `on_file_save_project_as` (`*.FlatPrj`) → `save_project()`. Serializes `{objs: [obj.to_dict()...], options, version}` to JSON, optionally LZMA-compressed (`global_save_compressed`, `global_compression_level`).
- **Script / Document** — `on_file_save_script` (`*.FlatScript`) and `on_file_save_document` (`*.FlatDoc`) write a `ScriptObject`/`DocumentObject` source via `save_source_file()`.
- **Preferences** — `on_export_preferences` (`*.FlatConfig`) writes the `options` dict; `on_file_save_defaults` persists defaults to `current_defaults.FlatConfig`.

### Open/Save dispatch mechanism

The dispatch lives in the `appIO` `QObject` (`appHandlers/appIO.py`), wired to the `File` menu. A typical flow:

1. The menu callback (`on_file_open_*` / `on_file_export_*`) shows a `QFileDialog` (open) or `FCFileSaveDialog` (save) with a format-specific filter string, remembering the last-used folder and filter.
2. For opens/exports it emits `self.worker_task.emit({'fcn': <handler>, 'params': [...]})` so the heavy parsing runs on a background `Worker` thread (the worker pool from `appWorkerStack.py`); project *open* is the exception and runs synchronously because it mutates app state non-thread-safely.
3. The handler wraps work in `self.app.proc_container.new(...)` (status-bar lifecycle) and calls `app.app_obj.new_object(kind, name, obj_init, ...)`, where `obj_init(obj, app_obj)` performs the actual `parse_file()`/import and returns `"fail"`/`"defective"` on error.
4. On success the file is registered as recently opened/saved via the `app.file_opened` / `app.file_saved` Qt signals, and a localized `[success]`/`[ERROR_NOTCL]` message is emitted through `self.inform`. ## Preprocessors (G-code Post-Processors)

The `preprocessors/` folder defines the G-code dialects FlatCAM emits. Each preprocessor is a thin class that turns the abstract operations of a CNC job (rapid move, plunge, lift, tool change, spindle/laser on-off, end) into the exact text strings a particular controller expects. The `CNCjob` engine in `camlib.py` walks the toolpath and calls these methods, splicing their return values together into the final G-code file.

### Base classes and registration (`appPreProcessor.py`)

Two abstract base classes, both built with the `ABCPreProcRegister` metaclass:

- **`PreProc`** — for milling (Geometry) and drilling (Excellon) jobs. Abstract methods include `start_code` (file header), `lift_code` (raise to safe Z), `down_code` (plunge to cut depth), `up_to_zero_code` (partial retract to material top), `rapid_code` (G0 travel), `linear_code` (G1 cut move), `toolchange_code`, `feedrate_code`, `spindle_code`, `spindle_stop_code`, and `end_code` (park move). Most also implement helper methods such as `startz_code`, `position_code` (X/Y formatting plus bed-skew/offset compensation), and `dwell_code`.
- **`AppPreProcTools`** — for the SolderPaste plugin, whose motion model is different. It exposes paste-specific hooks: `down_z_start_code`, `lift_z_dispense_code`, `down_z_stop_code`, `feedrate_xy_code`, `z_feedrate_code`, `feedrate_z_dispense_code`, `spindle_fwd_code`/`spindle_rev_code`/`spindle_off_code`, and `dwell_fwd_code`/`dwell_rev_code`.

**Auto-registration:** the `ABCPreProcRegister` metaclass (a subclass of `ABCMeta`) registers every concrete subclass into the module-global `preprocessors{}` dict at class-definition time, keyed by class name with an instantiated object as the value. Defining the subclass is all that is required — there is no manual registry. `load_preprocessors(app)` uses `importlib.machinery.SourceFileLoader` to glob and import every `*.py` under both the user data path's `preprocessors/` and the bundled `preprocessors/`, so users can drop in their own files. Re-registering a name logs an override warning. The class name must match the file name; for Roland files the name must contain the case-sensitive keyword `Roland`. Each preprocessor receives a parameter dict `p` (with attribute and key access) carrying units, coordinates, feedrates, Z heights, tool info, object options, decimals, and laser settings.

### Milling / Drilling controllers (subclass `PreProc`)

- **default** — Reference preprocessor; MACH3-compatible motion controllers. Emits a verbose comment header, `G20/G21 G90 G94`, `M6` tool changes with `M0` pause.
- **Default_no_M6** — Same MACH3-style output as `default` but omits the `M6` tool-change command (for controllers that choke on M6).
- **Berta_CNC** — Tailored for the BERTA CNC router.
- **ISEL_CNC** — For ISEL CNC routers.
- **ISEL_ICP_CNC** — For ISEL ICP CNC routers; semicolon-comment dialect.
- **NCCAD9** — Emits G-code for the Max Computer GmbH nccad9 controller.
- **Marlin** — For 3D-printer-style controllers running Marlin firmware (semicolon comments, Marlin G-code conventions).
- **Repetier** — For controllers running Repetier firmware.
- **GRBL_11** — Generic GRBL preprocessor, compatible with virtually any GRBL firmware version.
- **GRBL_11_no_M6** — Same as GRBL_11 but with no `M6` tool-change command.
- **Line_xyz** — Outputs every move with X, Y and Z on a single line (combined-coordinate format) rather than separate lift/down lines.
- **Check_points** — Diagnostic/positioning aid: moves a probe (e.g. a fine drill bit) to custom points and pauses for user interaction to verify PCB registration, rather than cutting.

### Laser variants (subclass `PreProc`, no Z cutting moves)

These assume a manually focused laser, move only in the X-Y plane, and toggle the beam via spindle/laser commands instead of plunging.

- **default_laser** — Default laser preprocessor for MACH3-compatible controllers; horizontal-only moves, manually focused laser.
- **GRBL_laser** — GRBL firmware with a laser on the SPINDLE connector; beam on with `M3`/`M4` (`laser_on_code`), off with `M5`, supports minimum-power idle.
- **GRBL_laser_z** — GRBL laser variant that does perform Z-axis moves (laser on the SPINDLE connector but with Z control retained).
- **GRBL_laser_air_assist** — GRBL 1.1 controllers (CNC 3018 and similar) with a laser module plus air-assist on the SPINDLE connector.
- **grbl_laser_eleks_drd** — GRBL laser cutters with no Z axis; converts Excellon drill codes into small arcs (requires controller `G2` support) so etched holes help center a drill bit for later manual drilling.
- **Marlin_laser_Spindle_pin** — Marlin firmware laser driven through the SPINDLE connector pin.
- **Marlin_laser_FAN_pin** — Marlin firmware laser driven through one of the FAN pins.
- **Marlin_laser_z** — Marlin firmware laser on the SPINDLE connector but retaining Z-axis moves.

### SolderPaste variants (subclass `AppPreProcTools`)

Used exclusively by the SolderPaste dispensing plugin; model nozzle dispense start/stop, dwell, and forward/reverse spindle (auger) control.

- **Paste_1** — SolderPaste dispensing for MACH3-like controllers.
- **Paste_GRBL** — SolderPaste dispensing for GRBL controllers (MACH3-like G-code conventions).
- **Paste_Marlin** — SolderPaste dispensing for Marlin-like controllers.

### Manual / Probe tool change (subclass `PreProc`)

- **Toolchange_Manual** — Special manual tool-change sequence; MACH3-compatible.
- **Toolchange_Probe_MACH3** — MACH3 tool change with automatic tool-length probing (touch-plate height measurement) between tools.

### Roland (subclass `PreProc`)

Output a Roland-specific command dialect (`^IN;`, `^PA;`, `SP`/pen-style codes) rather than standard G-code; class names contain the required `Roland` keyword.

- **Roland_MDX_20** — For the Roland MDX-20 desktop mill (1-decimal coordinate format, no standard header).
- **Roland_MDX_540** — For the Roland MDX-540 mill.

### HPGL (subclass `PreProc`)

- **hpgl** — Emits HPGL plotter language (`IN;`, `PU;`/`PD;` pen up/down, `PA` absolute moves, `SP` pen select) for HPGL-driven pen plotters/cutters. Works only in metric, scaling coordinates to 0.025 mm units and clamping to the valid HPGL coordinate range.

## Preferences & Settings

FlatCAM Evo centralizes all configurable behavior in a single, flat settings dictionary that is shared between the application defaults, the Preferences UI, and every object/tool. The system has three pieces: the `factory_defaults` baseline, the reactive `LoudDict` instances that hold live values, and an on-disk JSON persistence layer.

### Preferences system overview

- **`factory_defaults` (`defaults.py`)** — `AppDefaults.factory_defaults` is a single large dict (hundreds of keys) defining every setting and its baseline value. Keys follow a flat namespace convention by domain:
  - `global_*` — application-wide settings (e.g. `global_graphic_engine`, `global_appearance`, `global_worker_number`, `global_tolerance`, `global_systray_icon`, selection/highlight colors).
  - `units`, `units_precision`, `fit_key`, `zoom_in_key`, etc. — top-level constants and keyboard shortcuts.
  - `<kind>_<setting>` — per-object-type settings, where kind is `gerber`, `excellon`, `geometry`, `cncjob` (e.g. `gerber_circle_steps`, `excellon_drillz`, `geometry_cutz`).
  - `tools_<tool>_<setting>` — per-plugin settings (e.g. `tools_iso_*`, `tools_ncc_*`, `tools_paint_*`, `tools_mill_*`, `tools_drill_*`).
  - A `version` key (currently `8.992`, independent of the app version) drives migration logic.
- **`LoudDict` (`appCommon/Common.py`)** — `AppDefaults.defaults` is a `LoudDict`, a `dict` subclass whose `__setitem__` invokes a registered change callback (`set_change_callback`) after every write. This is what makes settings *reactive*: changing a value fires a callback that drives UI/state updates. Each `FlatCAMObj` likewise stores its per-object settings in an `obj_options` `LoudDict`, seeded at creation time from the app defaults whose keys are prefixed with the object's kind, so editing a field in an object's properties panel propagates live.
- **`AppDefaults` wrapper** — wraps the `LoudDict` with pass-through magic methods plus `current_defaults` (a plain-dict snapshot used to restore values when a Preferences edit is cancelled) and `propagate_defaults`/`reset_to_factory_defaults` helpers.
- **Persistence location** — the data folder is `%APPDATA%\FlatCAM` on Windows and `~/.FlatCAM` elsewhere (resolved in `appMain.py`; `config/` is used instead when `portable=True`). Settings are stored as JSON (`simplejson`) in version-stamped files:
  - `current_defaults_<version>.FlatConfig` — the live user settings (`AppDefaults.write`/`load`).
  - `factory_defaults_<version>.FlatConfig` — a written copy of the factory baseline.
  - `recent.json` / `recent_projects.json`, `log.txt`, and the `preprocessors/` and `tools_db_<version>.FlatDB` files also live here.
  - On load, `__is_old_defaults` compares the stored `version` against `factory_defaults['version']`; mismatches trigger migration (`__migrate_old_defaults`, type-checked key copy) in release builds, or a reset to factory defaults in beta builds.

### Preference categories (Preferences UI tabs)

The Preferences dialog is assembled under `appGUI/preferences/` from `OptionUI`/`OptionsGroupUI` building blocks (`PreferencesSectionUI`, `PreferencesUIManager`). Each top-level `*PreferencesUI` widget is a tab, composed of several `*PrefGroupUI` group panels:

- **General** (`general/GeneralPreferencesUI.py`) — App preferences (`GeneralAppPrefGroupUI`: units & precision, languages, worker/process counts, autosave, version check, send-stats, tolerance, PDF margins), GUI preferences (`GeneralGUIPrefGroupUI`: appearance/theme, dark canvas, layout, selection/highlight colors and shapes), and App settings (`GeneralAppSettingsGroupUI` / `GeneralAPPSetGroupUI`: grid behavior, mouse, notebook/workspace, shell).
- **Gerber** (`gerber/`) — General (`GerberGenPrefGroupUI`: plot, colors, circle steps), Options (`GerberOptPrefGroupUI`: isolation/non-copper/bbox defaults), Advanced Options (`GerberAdvOptPrefGroupUI`), Export (`GerberExpPrefGroupUI`: units/zeros/format on RS-274X export), and Editor (`GerberEditorPrefGroupUI`).
- **Excellon** (`excellon/`) — General, Options, Advanced Options, Export (`ExcellonExpPrefGroupUI`: format, units, zero suppression), and Editor groups — controlling drill/slot parsing, plotting, and drilling-job defaults.
- **Geometry** (`geometry/`) — General, Options (cut Z, travel Z, feedrates, multidepth, tool dia), Advanced Options, Export, and Editor groups for milling/geometry objects.
- **CNC Job** (`cncjob/`) — General (`CNCJobGenPrefGroupUI`: plot, annotation, tooldia display), Options, Advanced Options, Preprocessors (`CNCJobPPGroupUI`: G-code dialect selection), and Editor (`CNCJobEditorPrefGroupUI`) for generated G-code.
- **Tools / Plugins** (`tools/`) — the largest section, split into `PluginsPreferencesUI`, `Plugins2PreferencesUI`, and `PluginsEngravingPreferencesUI`. It holds one `*PrefGroupUI` per plugin, including: Isolation (ISO), Mill, Drill, NCC, Paint, Cutout, Film, Panelize, 2-Sided, Solderpaste, Transform, Calculators, Markers, Levelling, Substract (Sub), and the "Tools 2" set — Copper Thieving, Extract, Fiducials, Invert, Optimal, Punch Gerber, QRCode, and Rules Check. Each group exposes that plugin's default parameters (`tools_<tool>_*`).
- **Utilities** (`utilities/`) — `UtilPreferencesUI` with autocomplete keywords (`AutoCompletePrefGroupUI`) and the file-association preprocessor/keyword tables for Excellon, G-code, and Gerber (`FAExcPrefGroupUI`, `FAGcoPrefGroupUI`, `FAGrbPrefGroupUI`).

`PreferencesUIManager` binds each form widget to its `factory_defaults` key (defaults_form_fields mapping), reads the current values into the widgets, and writes edited values back into the `LoudDict` on Apply/OK (with `current_defaults` used to revert on Cancel).

### Tools Database (ToolsDB2)

`appDatabase.py` implements **ToolsDB2** (`ToolsDB2UI` plus the manager), a reusable library of named tool definitions persisted to `tools_db_<version>.FlatDB` (JSON) in the data folder. It is shared by the milling/drilling/isolation/paint/NCC/cutout plugins so a tool can be defined once and reused across jobs.

- **Structure** — `db_tool_dict` maps a tool id to a record of the form `{ 'tooldia': <float>, 'data': { ... } }`. The `tooldia` is the tool diameter; `data` holds the full parameter set (feedrates, cut/travel Z, multidepth, spindle speed, passes, overlap, offset, job/tool-type, etc.).
- **Target-typed tools** — each entry carries `data['tool_target']` (one of *General, Milling, Drilling, Isolation, Paint, NCC, Cutout*). When the target changes, the UI prunes the `data` keys that do not apply to that target, so a Drilling tool stores only drilling-relevant parameters, an NCC tool only NCC parameters, and so on. The tree view groups/sorts tools by target and diameter.
- **UI** — a three-column `FCTree` (ID, Name, Target, Diameter) on the left with parameter group boxes (Tool Description, Milling, and target-specific sections) on the right; tools are added/duplicated/deleted and edited there, with changes written back into `db_tool_dict` and saved to the `.FlatDB` file.
- **Usage** — plugins load tools from this database (matching by target type and diameter), seeding their per-tool `data` with the stored parameters, which then flow into the object's `obj_options` and ultimately into G-code generation.
